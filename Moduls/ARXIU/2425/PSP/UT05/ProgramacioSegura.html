    Utilització de tècniques de programació segura¶

    Introducció¶

    En general, quan s'envia alguna cosa a través de sòcols s'envia com a «text pla», és a dir, no sabem si hi ha algú fent servir un sniffer a la xarxa i per tant no sabem si algú està capturant les dades.

    En general, qualsevol sistema que pretengui ser segur necessitarà usar xifrat.

    Pràctiques de programació segura.¶

    Per enviar missatges xifrats es necessita algun mecanisme o algoritme per convertir un text normal en un de més difícil de comprendre.

    L'esquema general de tots els mètodes és tenir codi com el següent:

    public String cifrar  (String texto, String clave)  
{  
}

public String descifrar(String texto,String clave)
{
}

    Mètode Cèsar¶

    Si l'alfabet és el següent:

    ABCDEFGHIJKLMNÑOPQRSTUVWXYZ0123456789-

    BCDEFGHIJKLMNÑOPQRSTUVWXYZ0123456789-A

    El missatge HOLA MUNDO, amb clau 1 seria així

    HOLAMUNDO IPMBNVÑEP

    Criptografia de clau pública i clau privada.¶

    Els principals sistemes moderns de seguretat utilitzen dues claus d'una per xifrar i una altra per desxifrar. Això es pot fer servir de diverses formes.

    Principals aplicacions de la criptografia.¶

    • Missatgeria segura: tothom dona la seva clau de xifrat però conserva la de desxifrat. Si volem enviar un missatge a algú agafem la seva clau de xifrat i xifrem el missatge que li enviem. Només ell podrà desxifrar-lo.
    • Signatura digital: pilar del comerç electrònic. Permet verificar que un arxiu no ha estat modificat.
    • Missatgeria segura: en aquest tipus de missatgeria s'intenta evitar que un atacant (potser amb un sniffer) aconsegueixi desxifrar els nostres missatges.
    • Autenticació: els sistemes d' autenticació intenten resoldre una qüestió clau en la informàtica: verificar que una màquina és qui diu ser

    Protocols criptogràfics.¶

    En realitat protocols criptogràfics n'hi ha molts, i solen dividir-se en sistemes simètrics o asimètrics.

    • Els sistemes simètrics són aquells basats en una funció que converteix un missatge en un altre missatge xifrat. Si es vol desxifrar alguna cosa s'aplica el procés invers amb la mateixa clau que es va utilitzar.
    • Els sistemes asimètrics utilitzen una clau de xifrat i una altra de desxifrat. Encara que es tingui una clau és matemàticament impossible esbrinar l'altra clau pel que es pot donar a tothom una de les claus (anomenada habitualment clau pública) i conservar l'altra (anomenada clau privada). A més, podem fer servir les claus per al que vulguem i per exemple en uns casos xifrarem amb la clau pública i en d'altres tal vegada cifrem amb la clau privada.

    Ara per ara, les majors garanties les ofereixen els asimètrics, dels quals hi ha diversos sistemes. L'inconvenient que poden tenir els asimètrics és que són més lents computacionalment.

    En aquest curs farem servir el xifrat asimètric RSA.

    Encriptació d' informació.¶

    El següent codi mostra com crear una classe que permeti xifrar i desxifrar textos.

    public class GestorCifrado {
    KeyPair claves;
    KeyPairGenerator generadorClaves;
    Cipher cifrador;
    public GestorCifrado()
        throws NoSuchAlgorithmException,
    NoSuchPaddingException{
    generadorClaves=
        KeyPairGenerator.getInstance("RSA");
    /*Usaremos una longitud de clave
     * de 1024 bits */
    generadorClaves.initialize(1024);
    claves=generadorClaves.generateKeyPair();
    cifrador=Cipher.getInstance("RSA");
    }
    public PublicKey getPublica(){
    return claves.getPublic();
    }
    public PrivateKey getPrivada(){
    return claves.getPrivate();
    }

    public byte[] cifrar(byte[] paraCifrar,
        Key claveCifrado
        ) throws InvalidKeyException,
        IllegalBlockSizeException,
        BadPaddingException{
    byte[] resultado;
    /* Se pone el cifrador en modo cifrado*/
    cifrador.init(Cipher.ENCRYPT_MODE,
        claveCifrado);
    resultado=cifrador.doFinal(paraCifrar);
    return resultado;
    }

    public byte[] descifrar(
        byte[] paraDescifrar,
        Key claveDescifrado)
            throws InvalidKeyException,
            IllegalBlockSizeException,
            BadPaddingException{
    byte[] resultado;
    /* Se pone el cifrador en modo descifrado*/
    cifrador.init(Cipher.DECRYPT_MODE,
        claveDescifrado);
    resultado=cifrador.doFinal(paraDescifrar);
    return resultado;
    }



    public static void main(String[] args)
        throws NoSuchAlgorithmException,
        NoSuchPaddingException,
        InvalidKeyException,
        IllegalBlockSizeException,
        BadPaddingException,
        UnsupportedEncodingException
    {
    GestorCifrado gestorCifrado=
        new GestorCifrado();
    String mensajeOriginal="Hola mundo";
    Key clavePublica=gestorCifrado.getPublica();

    byte[] mensajeCifrado=
        gestorCifrado.cifrar(
            mensajeOriginal.getBytes(),
            clavePublica
    );
    String cadCifrada=
        new String(mensajeCifrado, "UTF-8");

    System.out.println
        ("Cadena original:"+mensajeOriginal);
    System.out.println
        ("Cadena cifrada:"+cadCifrada);

    /* Cogemos la cadCifrada y la desciframos
     * con la otra clave */
    Key clavePrivada;
    clavePrivada=gestorCifrado.getPrivada();
    byte[] descifrada=
        gestorCifrado.descifrar(
            mensajeCifrado,clavePrivada);
    String mensajeDescifrado;
    /* E imprimimos el mensaje*/
    mensajeDescifrado=
        new String(descifrada, "UTF-8");
    System.out.println(
        "El mensaje descifrado es:"+
            mensajeDescifrado);
    }
}

    Treballant amb blocs llargs¶

    En general els diferents mètodes criptogràfics no sempre poden treballar amb blocs tot el llargs que vulguem, així que si es necessita «xifrar una cadena llarga» és molt probable que haguem d'anar «xifrant bloc a bloc».

    Protocols segurs de comunicacions.¶

    En general, ara que ja coneixem sockets, l'ús de servidors i clients i l'ús de la criptografia de clau asimètrica ja és possible crear aplicacions que es comuniquin de forma molt segura.

    En general, tot protocol que vulguem implementar farà aquests passos.

    1. Tot client genera la seva parella de claus.
    2. Tot servidor genera la seva parella de claus.
    3. Quan un client es connecti a un servidor, li envia la seva clau de xifrat i conserva la de desxifrat.
    4. Quan un servidor rep la connexió d' un client rep la clau de xifrat d' aquest client.
    5. El servidor envia la seva clau pública al client.
    6. Ara client i servidor poden enviar missatges a l'altre amb la garantia que només servidor i client respectivament poden desxifrar.

    En realitat es pot assegurar més el procés fent que en el pas 5 el servidor cifre la seva pròpia clau pública amb la clau pública del client. D'aquesta manera, fins i tot encara que algú robés la clau privada del client tampoc tindria massa, ja que hauria de robar la clau privada del servidor.

    Infraestructura de clau pública (PKI)¶

    Per garantir la seguretat cal que entri un tercer jugador en l' intercanvi de claus entre clients i servidors. Aquest tercer individu són les autoritats de certificació .

    Programació d' aplicacions amb comunicacions segures.¶

    Per fortuna Java disposa de classes ja prefabricades que faciliten enormement el que dues aplicacions de forma segura a través d'una xarxa. S' han de considerar els punts següents:

    • El servidor ha de tenir el seu propi certificat. Si no el tenim, es pot generar primer una parella de claus amb l'eina keytool, com es mostra en la figura adjunta. L'eina guardarà la parella de claus en un magatzem (el qual té la seva pròpia clau). Després generarem un certificat a partir d'aquesta parella amb keytool -export -file certificadoservidor.cer -keystore magatzems.
    • El codi del servidor necessitarà indicar el fitxer on s' emmagatzemen les claus i la clau per accedir a aquest magatzem.
    • El client necessita indicar que confia en el certificat del servidor. Aquest certificat del servidor pot estar guardat (per exemple) al magatzem de claus del client.
    • Tot i que no sol fer-se també podria fer-se a la inversa i obligar el client a tenir un certificat que el servidor pogués importar, cosa que augmentaria la seguretat.

    Els passos desglossats impliquen executar aquests comandaments en el servidor:

    # El servidor genera una pareja de claves que se almacena en un
#fichero llamado "clavesservidor". Dentro del fichero se indica
#un alias para poder referirnos a esa clave fácilmente
keytool -genkeypair -keyalg RSA
     -alias servidor -keystore clavesservidor

#El servidor genera su "certificado", es decir un fichero que
#de alguna forma indica quien es él. El certificado se almacena
#en un fichero llamado clavesservidor y a partir de él queremos
#generar el certificado de un alias que tiene que haber llamado servidor
keytool --exportcert -alias servidor
     -file servidor.cer -keystore clavesservidor

    En el client donarem aquests passos:

    #Se genera una pareja de claves (en realidad no nos hace falta solo
#queremos tener un almacén de claves.
keytool -genkeypair -keyalg RSA -alias cliente -keystore clavescliente

#Se importa el certificado del servidor indicando que pertenece a
#la lista de certificados confiables.
keytool -importcert -trustcacerts -alias servidor -file servidor.cer -keystore clavescliente

    Un cop creats els fitxers inicials s'han de donar els següents passos a Java (servidor i client van per separat):

    1. El servidor ha de carregar el seu magatzem de claus (el fitxer clausservidor)
    2. Aquest magatzem (carregat en un objecte Java anomenat KeyStore), es farà servir per crear un gestor de claus (objecte KeyManager), el qual s'obté a partir d'una «fàbrica» anomenada KeyManagerFactory.
    3. Es crearà un context SSL (objecte SSLContext) a partir de la fàbrica comentada.
    4. L'objecte SSLContext permetrà crear una fàbrica de sockets que serà la que finalment ens permeti tenir un SSLServerSocket, és a dir un socket de servidor que farà servir xifrat.

    El codi Java del servidor seria una cosa així: 

    public OtroServidor (String rutaAlmacen, String claveAlmacen){
        this.rutaAlmacen=rutaAlmacen;
        this.claveAlmacen=claveAlmacen;
    }

    public SSLServerSocket getServerSocketSeguro()
            throws KeyStoreException, NoSuchAlgorithmException,
            CertificateException, IOException,
            KeyManagementException, UnrecoverableKeyException
    {
        SSLServerSocket serverSocket=null;
        /* Paso 1, se carga el almacén de claves*/
        FileInputStream fichAlmacen=
                new FileInputStream(this.rutaAlmacen);
        /* Paso 1.1, se crea un almacén del tipo por defecto
         * que es un JKS (Java Key Store), a día de hoy*/
        KeyStore almacen=KeyStore.getInstance(KeyStore.getDefaultType());
        almacen.load(fichAlmacen, claveAlmacen.toCharArray());
        /* Paso 2: obtener una fábrica de KeyManagers que ofrezcan
         * soporte al algoritmo por defecto*/
        KeyManagerFactory fabrica=
                KeyManagerFactory.getInstance(
                KeyManagerFactory.getDefaultAlgorithm());
        fabrica.init(almacen, claveAlmacen.toCharArray());
        /* Paso 3:Intentamos obtener un contexto SSL
         * que ofrezca soporte a TLS (el sistema más
         * seguro hoy día) */
        SSLContext contextoSSL=SSLContext.getInstance("TLS");
        contextoSSL.init(fabrica.getKeyManagers(), null, null);
        /* Paso 4: Se obtiene una fábrica de sockets que permita
         * obtener un SSLServerSocket */
        SSLServerSocketFactory fabricaSockets=
                contextoSSL.getServerSocketFactory();
        serverSocket=
            (SSLServerSocket)
                fabricaSockets.createServerSocket(puerto);
        return serverSocket;
    }
    public void escuchar()
    throws KeyManagementException, UnrecoverableKeyException,
        KeyStoreException, NoSuchAlgorithmException,
        CertificateException, IOException
    {
        SSLServerSocket socketServidor=this.getServerSocketSeguro();
        BufferedReader entrada;
        PrintWriter salida;
        while (true){
            Socket connRecibida=socketServidor.accept();
            System.out.println("Conexion segura recibida");
            entrada=
            new BufferedReader(
            new InputStreamReader(connRecibida.getInputStream()));
            salida=
            new PrintWriter(
                new OutputStreamWriter(
                connRecibida.getOutputStream()));
            String linea=entrada.readLine();
            salida.println(linea.length());
            salida.flush();
        }
    }

    En el client s' han de fer alguns passos semblants:

    1. En primer lloc es carrega el magatzem de claus del client (que conté el certificat del servidor i que és la clau per poder «autenticar» el servidor)
    2. El magatzem del client es farà servir per crear un «gestor de confiança» (TrustManager) que Java farà servir per determinar si pot confiar o no en una connexió. Usarem un TrustManagerFactory que farà servir el magatzem del client per crear objectes que puguin gestionar la confiança.
    3. Es crearà un context SSL (SSLContext) que es basarà en els TrustManager que pugui crear la fàbrica.
    4. A partir del context SSL el client ja pot crear un socket segur (SSLSocket) que pot fer servir per connectar amb el servidor de forma segura.

    El codi del client seria una cosa així:

    public class OtroCliente {
    String almacen="/home/usuario/clavescliente";
    String clave="abcdabcd";
    SSLSocket conexion;
    public OtroCliente(String ip, int puerto)
            throws UnknownHostException, IOException,
            KeyManagementException, NoSuchAlgorithmException,
            KeyStoreException, CertificateException{

        conexion=this.obtenerSocket(ip,puerto);
    }
    /* Envía un mensaje de prueba para verificar que la conexión
     * SSL es correcta */
    public void conectar() throws IOException{
        System.out.println("Iniciando..");
        BufferedReader entrada;
        PrintWriter salida;
        entrada=new BufferedReader(new InputStreamReader(conexion.getInputStream()));
        salida=new PrintWriter(new OutputStreamWriter(conexion.getOutputStream()));
        /* De esta linea se intenta averiguar la longitud*/
        salida.println("1234567890");
        salida.flush();
        /* Si todo va bien, el servidor nos contesta el numero*/
        String num=entrada.readLine();
        int longitud=Integer.parseInt(num);
        System.out.println("La longitud devuelta es:"+longitud);

    }
    public SSLSocket obtenerSocket(String ip, int puerto)
            throws KeyStoreException, NoSuchAlgorithmException,
            CertificateException, IOException, KeyManagementException
    {
        System.out.println("Obteniendo socket");
        SSLSocket socket=null;
        /* Paso 1: se carga al almacén de claves
         * (que recordemos debe contener el
         * certificado del servidor)*/
        KeyStore almacenCliente=KeyStore.getInstance(KeyStore.getDefaultType());
        FileInputStream ficheroAlmacenClaves=
                new FileInputStream( this.almacen );
        almacenCliente.load(ficheroAlmacenClaves, clave.toCharArray());
        System.out.println("Almacen cargado");
        /* Paso 2, crearemos una fabrica de gestores de confianza
         * que use el almacén cargado antes (que contiene el
         * certificado del servidor)
         */
        TrustManagerFactory fabricaGestoresConfianza=
                TrustManagerFactory.getInstance(
                        TrustManagerFactory.getDefaultAlgorithm());
        fabricaGestoresConfianza.init(almacenCliente);
        System.out.println("Fabrica Trust creada");
        /*Paso 3: se crea el contexto SSL, que ofrezca
         * soporte al algoritmo TLS*/
        SSLContext contexto=SSLContext.getInstance("TLS");
        contexto.init(
                null, fabricaGestoresConfianza.getTrustManagers(), null);
        /* Paso 4: Se crea un socket que conecte con el servidor*/
        System.out.println("Contexto creado");
        SSLSocketFactory fabricaSockets=
                contexto.getSocketFactory();
        socket=(SSLSocket) fabricaSockets.createSocket(ip, puerto);
        /* Y devolvemos el socket*/
        System.out.println("Socket creado");
        return socket;
    }
}

    Signat d' aplicacions¶

    Utilitzant la criptografia de clau pública és possible «signar» aplicacions. El signat és un mecanisme que permet a l'usuari d'una aplicació el verificar que l'aplicació no ha estat alterada des que el programador la va crear (virus o programes malignes, personal descontent amb l'empresa, etc...).

    Abans d' efectuar el signat s' ha de disposar d' un parell de claus generades amb l' eina keytool esmentada anteriorment. Suposem que el magatzem de claus està creat i que hi ha un o diversos àlies creats . El procés de signat és el següent:

    1. Crear l' aplicació, que pot estar formada per un conjunt de classes però que en última instància tindrà un main.
    2. Empaquetar l'aplicació amb eixe Aplicacion.jar com.ies.Aplicacion *. Aquest comandament crea un fitxer (f) JAR en el qual el punt d'entrada (e) és la classe com.ies.Aplicacion (que és la que tindrà el main).
    3. Es pot comprovar que l'aplicació dins del JAR s'executa correctament amb java -jar Aplicacion.jar.
    4. Ara es pot executar jarsigner -keystore <ruta-magatzem> Aplicacion.jar <àlies>.

    Amb aquests passos es té una aplicació signada que l'usuari pot verificar si així ho desitja. De fet, si s'extreu el contingut del JAR amb jar -xf Aplicacion.jar s'extreuen els arxius .class i un fitxer META-INF/Manifest que es pot obrir amb un editor per veure que realment està signat.

    Perquè altres persones puguin comprovar que la nostra aplicació és correcta els programadors haurem d'exportar un certificat que els usuaris puguin importar per fer el verificat. Recordem que el comandament és:

    keytool -exportcert -keystore ...\Magatzem.store -file Programador.cer -àlies Programador

    Verificat d' aplicacions¶

    Si ara un altre usuari desitja executar la nostra aplicació haurà d'importar el nostre certificat. El procés de verificat és simple:

    1. L'usuari importa el certificat.
    2. Ara que té el certificat pot comprovar l'aplicació amb jarsigner -verify -keystore <ruta-magatzem> Aplicacion.jar <alias_del_programador>

    El comandament haurà de respondre amb alguna cosa com ara jar verified. Tanmateix si no tenim un certificat signat per alguna autoritat de certificació (CA) l'eina es queixarà que alguns criteris de seguretat no es compleixen.

    Recordatori¶

    Hem fet el procés de signat i verificat amb certificats autofirmats, la qual cosa és útil per practicar però completament inútil des del punt de vista de la seguretat. Perquè un certificat sigui segur hem de fer que prèviament alguna autoritat de certificació ens ho signi primer (per a la qual cosa sol ser habitual haver de pagar).

    Política de seguretat.¶

    Java inclou un mecanisme per definir polítiques de seguretat. La definició oficial de Java per a una política és objecte que especifica quins permisos estan disponibles per al codi en funció del seu origen i de l'usuari amb el qual s'executen. Aquest origen pot ser els diversos directoris del sistema operatiu o fins i tot adreces URL.

    Cal destacar que tot el que s'esmenta aquí no funciona si l'usuari té accés al sistema i pot executar l'intèrpret de Java sense restriccions. És a dir, es necessita el treball d' un administrador de sistemes per restringir la manera com l' usuari executa el codi.

    Suposem llavors que estem en un entorn segur on els programes Java s'executen utilitzant un gestor de seguretat, és a dir, es llancen executant java -Djava.security.manager Classe. En principi, els programes no podran fer gaires coses, com per exemple, connectar-se a Internet o llegir un fitxer que no estigui en el mateix directori de la classe.
